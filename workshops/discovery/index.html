Journey - Discovery

This is the current outline for the portion of our general API Journey workshops focused on understanding where your digital resources are is critical to maturing and evolving in how you do APIs. API discovery means many different things to different groups, but in the context of this journey, it is about knowing where your APIs are, and understanding what your organization is capable of doing. Having a coherent catalog of API definitions for ALL of your organizational services, as well as the technical, and business details behind them.

Goals

Defining what the meaning and goals for API discovery are, acknowledging there are many different perspectives on what is API discovery, and what it involves to move the conversation around it forward.

  1. 1) New APIs - Being able to find new APIs when needed.
  2. 2) Application Development - Being able to discover and identify the right APIs to be using when building a specific application.
  3. 3) Team Planning - For use as part of wider team planning, and being able to articulate what is in production, what is being developed, and what is currently being defined and designed as part of a single group.
  4. 4) External - Being able to find APIs quickly, as well as provide self-service discovery of services by clients, partners, and other external actors.
  5. 5) Developing An API - A team is looking to develop an API and they want to understand if one already exists, or maybe better understand existing patterns in use by production applications.
  6. 6) Execution - Being able to find an API so that you can make a call to it in some code, or through a client application like Postman.
  7. 7) Governance - Being able to quantify what APIs exist across any group, organization, domain, and the enterprise enterprise at any moment.

Definitions

Work to ensure that all definitions are being aggregated as part of the process so that they can be evolved and moved forward into design, development and production--investing in all of the artifacts that will be needed down the road.

  1. 1) Teams - Map out all of the individuals being engaged as part of the service and API discovery process, grouping them by team, group, or other organizational structure.
  2. 2) Organizations - Defining the organizational structure, group, or other bounded context for organizing definitions into.
  3. 3) Domains - Considering the domain that services fits into and exploring the sphere of knowledge and activity around the domain each service is delivering value within. Quantifying the service domain, helping decompose from a larger domain, and helping focus what each service does.
  4. 4) Vocabularies - Organizing the common vocabulary in use across services, and APIs, establishing a single place to find the language used across domains.
  5. 5) Tags - Defining what tagging taxonomy will be applied across resources, for use in tagging across definitions, documentation, and infrastructure operations.
  6. 6) Event Storming - Event storming is a rapid, lightweight, and underappreciated group modeling technique that is intense, fun, and useful for accelerating development teams.
  7. 7) Schema - Organizing individual schema associated with a service, while also including it as part of the OpenAPI. Providing SQL dumps, JSON Schema, and other artifacts that help define the schema that are in use across the enterprise.
  8. 8) Contracts - Having a formal definition of each API that is depended on as part of operations. If it is driving business it should have a robust OpenAPI contract behind it, defining every aspect of the service it provides.
  9. 9) Headers - Establishing a master catalog of all HTTP headers, and identifying the ones that are in use. Understanding the overall header landscape across services, and how they are leverage by other stops along the API lifecycle like authentication, security, tracing, etc.
  10. 10) Parameters - Possessing a birds eye view of parameters used across all APIs, and having a commonly defined set of parameters that can be used and reused across APIs during their design, development, and ultimately production phases.
  11. 11) Servers - Documenting all of the servers in use across API infrastructure, and using their existing outputs to understand what APIs operate or are consumed there. Providing a central set of definitions regarding all server infrastructure which can be reused across API operations, and as part of the design, development, and production phases.
  12. 12) Environments - Having a common way to define, organize, share, and audit API environments that can be used as part of the development and operation of API infrastructure, as well as by consumer facing elements of operations. Using common formats like Postman Environments to define, and organize common environments that will exist across the enterprise.

Data Sources

Understanding what resources are already on the public websites, and applications, by spidering existing domains looking for data assets that should be delivered as API resources. Looking for the low hanging fruit of digital resources that already exists across an organization, which has already been published for a reason, and should most likely also be available as a machine readable API.

  1. 1) Spreadsheets - adfIdentifying all of the spreadsheet files that are regularly published and shared across internal groups, with partners and the public, ensuring that all spreadsheets are available as machine readable, as well as human readable sources of information.
  2. 2) Comma Separate Values (CSV) - Identify all of the CSV files that have been published within any domains.
  3. 3) Extensible Markup Language (XML) - Identify all of the XML files that have been published within any domains.
  4. 4) JavaScript Object Notation (JSON) - Identify all of the JSON files that have been published within any domains.
  5. 5) Tables - Identify all of the HTML tables that have been published within any domains.
  6. 6) HTML Forms - Looking at the usage of forms across websites, and other applications, to identify what the data source behind them are, and how they can be tapped as a potential API source, eventually replacing forms with better organized embeddable API tooling.

Low hanging fruit is about identifying the resources that already exist, are already being shared with internal groups, partners, and with the public via highly trafficked areas of the web. Actively understanding what is being published by groups across the enterprise, and helping incentivize all teams to publish all digital resources as API-first.

Discovery Sources

Looking for many of the common sources where signals of APIs can be found as part of the ongoing API discovery process. Helping automate and add to the API catalog beyond just the manual, self-declared processes that may occur.

  1. 1) Log Files - Looking at existing sources of log files, and to where log files are shipped to, examining them for API signals that can be harvested and used to build the API discovery catalogs.
  2. 2) AWS Cloudfront - Leveraging AWS Cloudfront to find existing signs of APIs in use.
  3. 3) Google Stackdriver - Leveraging Google Stackdriver to find existing signs of APIs in use.
  4. 4) Aure Logging - Leveraging Azure logging to find existing signs of APIs in use.
  5. 5) Email - Looking at email when it comes to discovery, allowing for easy sharing via common used messaging format, but also the ability to scan and forward emails for common API file formats.
  6. 6) GitHub - Looking for API signals that exist across GitHub repositories, user accounts, and organizations.

Discovery Formats

Defining the potential sources of web service and API signals that can be tapped from around the enterprise. Tuning into existing workspaces, content and data sources, looking for API signals that can be harvested. Helping map out the API landscape that already exists, using the outputs of existing systems that already operate as part of the API supply chain within the enterprise.

  1. 1) WSDL - Looking for WSDL documents across an organization.
  2. 2) API.json - Looking for APIs.json files across an organization.
  3. 3) OpenAPI - Looking for OpenAPI files across an organization.
  4. 4) Postman Collection - Looking for Postman Collections across an organization.
  5. 5) .HAR - Looking for evidence of APIs from .HAR files that are gathered from proxies, browsers, and other tooling, aggregating the signals gathered and using to map out the API landscape across an organization in real time.
  6. 6) JSON Schema - Looking for JSON schema files that can be used to help build out OpenAPI definitions, and provide the seeds for API contracts that may already be in use for applications, or as part of existing infrastructure and operations.
  7. 7) RAML - Looking for RAML files that already exist, or can be generated from out of existing systems.
  8. 8) API Blueprint - Looking for API Blueprint files that already exist, or can be generated from out of existing systems.

There are signals of API usage all over the enterprise. While it is ideal for API development teams to be able to submit the definitions of their APIs to a central catalog, it is something that should always be augmented with more automated approaches to identifying potential data sources, and scanning common locations to discover existing API usage in real time.

Dependencies

Working to define any dependencies that are in play, and will play a role in operations by auditing the stack behind any service as it is being discovered and documented as part of the overall API discovery effort.

  1. 1) Services - What existing services does a service depend on to do what it does, and would be impacted if something changes.
  2. 2) Software - What are the software libraries a service depends on for its operations, covering the code layer dependencies.
  3. 3) Data - What is the provenance of data being used as part of the service, and where are their continued dependencies.
  4. 4) People - What people is a service dependent on, requiring coordination, communication, and effort from individuals to move forward, and ensuring the right stakeholders are involved.
  5. 5) Organizations - What other organizations, internally or externally are required to move a service forward, making sure all stakeholders are considered.
  6. 6) Applications - Documenting the client applications dependencies for a service, outlining who depends on the service based upon the applications they are using.
  7. 7) Connections - Making sure to connect all the dots when it comes to mapping out dependencies, showing the relationships between services, software, data, people, organizations, and applications.

Catalog

Actively maintaining a catalog of all APIs in use, working to automatically identify new APIs in use, as well as actively profiling, and increasing access and usage of more refined, better known APIs across the enterprise.

  1. 1) Overview - Providing an overview of what the API catalog, and helping explain what the catalog is for, and how to use it properly.
  2. 2) Collections - Allow the organization of APIs into different types of collections, allowing individual, or groups of APIs and services to be organized into precise collections for use across the enterprise.
  3. 3) Definitions - Allow for the cataloging of multiple types of API definitions for any particular API, documenting OpenAPI, API Blueprint, RAML, WSDL, Postman Collections, and any other artifact that supports API discovery.
  4. 4) Data - Allowing for datasets to be registered with the catalog, adding regular datasets to the discovery effort, and make available alongside APIs, and the applications that depend on them.
  5. 5) Applications - Allowing applications to be registered in the catalog alongside datasets, and APIs, completing the loop when it comes to discovery across a platform.
  6. 6) Servers - Provide a central catalog of all servers that are used across API infrastructure, tapping all the definitions present for any domain, subdomain, and IP address in usage across the enterprise.
  7. 7) Search - Provide basic, as well as more advanced search allowing all APIs, artifacts, and other meta data to b searched, filtered, and sorted.
  8. 8) Self-Service - Ensuring that the API catalog is self-service, allowing all targeted providers, consumers, and other stakeholders to actively browse, search, and manage the API catalog in a self-service way.
  9. 9) Personalizaton - Allow for users of the catalog to personalize and customize their catalog experience by favoriting, labeling, and providing other ways to make it more useful for each individual using it.

Communication

Having a strategy for reaching out to teams and engaging with them around API discovery, helping them remember to register and define their APIs as part of wider strategy.

  1. 1) Email - Use email to coordinate with other groups regarding the APIs they are supporting, working on, and planning on designing in the near future.
  2. 2) Meetings - Provide a recurring set of meetings focused on discussing what APIs exist, and are being developed, ensuring that teams come together regularly to discuss where the APIs and services are.
  3. 3) Workshops - Conducting workshops to actively develop wider discovery practices, helping teams make their API catalogs more discoverable, and in alignment with other teams.
  4. 4) Strategy - Having a strategy for how APIs and other services are identified, aggregated, cataloged, and made available across the enterprise for everyone to put to work in an efficient way.

Support

Ensure that all teams have support when it comes to questions about web service and API discovery, helping them initially, as well as along the way, making sure all their APIs are accounted for, and indexed as part of discovery efforts.

  1. 1) Email - Having a dedicated email account for supporting each service.
  2. 2) GitHub Issues - Where to submit an issue to get support for each service.

Evangelism

Considering the evangelism around just API discovery, and helping amplify its effect. Actively telling stories around the platform, and keeping groups informed around what APIs are available, and the best practices around making them more discoverable and consumable.

  1. 1) New APIs - Have a new API widget that can live on the catalog home page, across other API portals, or any other web property helping surface any APIs that were just added to the catalog.
  2. 2) Active APIs - Have an active API widget that can live on the catalog home page, across other API portals, or any other web property helping surface any APIs that have been recently change, updated, or move into new phase.
  3. 3) Newsletter - Providing an email newsletter that regularly highlights new, active, and other APIs in the catalog. Also bringing attention to best practices, and other building blocks of API discovery across the enterprise.
  4. 4) Talks - Giving talks with other internal, partner, or public groups regarding the availability of APIs, and what discovery looks like across the enterprise, helping spread the word about what is happening.
  5. 5) Reporting - Providing a regular set of reports to stakeholders regarding the number of APIs, the type of APIs, by team, organization, and other metric. Delivering a regular set of reports that stakeholders can tune into to understand what is happening with API discovery, and the importance of it moving forward and evolving.
