Journey - Production

This is the current outline for the portion of our general API Journey workshops focused on managing APIs in a production environment, addressing what is needed to successfully operate and evolve your critical API infrastructure.

Definitions

All of the definitions available for each service, providing a single way to find everything that drives the project.

  1. 1) Organization - The over-arching source control organization for each service, providing an umbrella for different development groups and projects.
  2. 2) Domain - Defining which domain an API is operating under, ensuring that it is considered as part of the wider planning, and evolution within a specific domain.
  3. 3) Repository - The source control repository for each service, providing a self-contained, shareable, forkable way to engage with a service. Allowing for EVERYTHING involved with a service to be accessed in a single location.
  4. 4) Team - A list of team members for the service, providing a nice list of who is responsible for making a service operate, defining who ultimately owns and sustains each service.
  5. 5) README - The README for each services repository, providing an overview of the service, complete with all the relevant links to all the moving parts. Any user should be able to land on the README, and easily get up to speed on what a service delivers.
  6. 6) OpenAPI - An OpenAPI definition for the surface area of the API, providing a machine readable contract for what each service delivers, that can be used across the API lifecycle, guiding every aspect of delivering each service.
  7. 7) AsyncAPI - Defining how AsyncAPI definitions are used across the production environment to define event-driven capabilities
  8. 8) Postman Collections - Defining how Postman Collections are used across a production environment to engage with APIs, and orchestrate operations using APIs.
  9. 9) JSON Schema - Defining what JSON Schema is available for supporting the operations of an API, providing the machine readable definition of the underlying schema used.
  10. 10) Scopes - Defining what OAuth scopes are present for use as part of an API's operation, defining the access management layer for an API.
  11. 11) Tags - Using the common tags and taxonomy for operating infrastructure and other resources, allowing for easy discovery, measurement and reporting of what it takes to operate each service.

Providing a machine readable core for the entire lifecycle of each service, allowing documentation to be used elsewhere.

  1. 1) Spans - Defining the primary building blocks of a trace, representing an individual unit of work done in any system.
  2. 2) Scope - Providing a way of describing the activity that occurs across a single span within the tracing flow.
  3. 3) Tracers - The mechanisms for pushing or injecting a transaction to measure, qualify, and understand the tracing flow.
  4. 4) Tags - Having a vocabulary for labeling and organizing tracing operations, allowing them to be grouped and reported upon in meaningful ways that speaks to the rest of the API lifecycle, and maps to overall definitions and governance.
  5. 5) Trackers - Defining what is used to uniquely define each API call, tracers, and other transactions that are being quantified as part of overall tracing efforts.
