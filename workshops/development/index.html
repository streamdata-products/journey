API Development

This is the current outline for the portion of our general API Journey workshops focused on the rapid, efficient, and consistent design of APIs at scale across the enterprise.

Definitions

This is the current outline for the portion of our general API Journey workshops focused on the development process, making sure the API contract is used to drive th development of code, and moving it from development to QA, and into production.

  1. 1) Organization - Defining which organization each API belongs to while it is being designed, understanding which organization is responsible for it throughout this phase.
  2. 2) Domain - Defining which domain an API is being designed in, connecting the dots with domain experts, resources, and other relevant context that should be considered as an API is being iterated upon.
  3. 3) Repository - Use repository to version, build, and deploy each service.
  4. 4) Team - Extend the team list for a service to include the development, QA, and other stakeholders involved in this area of the lifecycle.
  5. 5) README - Update the README for the repository to reflect what is happening as part of development.
  6. 6) Schema - JSON schema for the data behind service, providing a standalone JSON schema for use in other systems, applications, and services. Doing the heavy lifting of describing the objects, fields, types, and other technical details of the data behind each service.
  7. 7) OpenAPI - An OpenAPI definition for the surface area of the API, providing a machine readable contract for what each service delivers, that can be used across the API lifecycle, guiding every aspect of delivering each service.
  8. 8) AsyncAPI - Defining how AsyncAPIs are used as part of the design process to document the event-driven characteristics of the platform.
  9. 9) Postman Collections - Defining how Postman Collections are used as part of the API design process, to help make APIs more executable, mockable, and usable during the design phase.
  10. 10) Tags - What tags are associated with an API, making it more discoverable and organizable as part of the larger API catalog.

Discovery

Making sure there are machine readable definitions available at development time, providing an OpenAPI as a contract for what each service should do.

  1. 1) Catalog - Ensuring that the central API catalog is leverage as part of the API design workflow. Pulling API definitions out and moving them forward, while always leaning on the catalog as the central truth.
  2. 2) Filter - Being able to filter APIs in the catalog by the phase they are in, understanding if they are currently being developed.
  3. 3) Search - Being able to search across APIs that are being developed by specific characteristics of the collections they are in, the OpenAPI contract, or other granular details.

Virtualization

Continuing to maintain virtualized instances of APIs as part of the API development process.

  1. 1) Paths - Offering mock API paths for use by developers to understand what an API does / should do.
  2. 2) Data - Considering offering synthetic data for use by integrators, helping ensure virtualized APIs provide as production like experience in a virtualized way.
  3. 3) Dedicated Sandbox - Adding an API to a dedicated sandbox ensuring that a virtualized instance is always available for consumers to play with and understand what is possible.
  4. 4) Personalized Sandbox - Providing Docker images, machine images, and other virtualized instances of an API, allowing consumers to deploy their own personal virtualization of an API.

Database

The database behind each service, acting as the data source for these resources.

  1. 1) Platform - The database platform used to drive each service, providing a dedicated database to just drive each service.
  2. 2) Schema - Making sure there is a database version of the schema, providing a raw dump and backup of the database behind each service.
  3. 3) Region - Where the database for each service is located, helping articulate if there is replication, or regional specific implementations or considerations for each service.
  4. 4) Access - How the database is connected with by the service, defining what connectivity is required, and what dependencies are involved with access.
  5. 5) Logs - Understanding how logging occurs for the database, what is turned on, and where logs are stored, and or shipped to as part of regular operations.
  6. 6) Backup - Ensuring that the database has a backup strategy, and the details of what is backed up and how long it is being backed up.
  7. 7) Encryption - Ensuring the encryption is part of the database operations, making sure data is being stored as part of the database behind each service.

Providing a dedicated database resource for just this single service to use.

Storage

The storage used for each service, where all objects are stored.

  1. 1) Platform - The platform that is used for storage behind each service.
  2. 2) Region - Where the storage is located that is used for each service.
  3. 3) Access - Details on how storage is accessed as part of each service.
  4. 4) Logs - Information about the logging for storage behind service.
  5. 5) Backup - Ensuring there is a backup strategy for storage behind service.
  6. 6) Encryption - Ensuring encryption is considered for storage.

Providing a single location each service to manage it's storage.

DNS

The DNS addressing used to route all traffic to each service.

  1. 1) Platform - Which platform is used to provide DNS for service.
  2. 2) Mock - The host used for the mock API.
  3. 3) Development - The host used for the development API, providing access to the version of API currently being developed.
  4. 4) Production - The host used for the production API, which is the address that should actually be used in applications.
  5. 5) Portal - The host used for the developer portal, providing a single location that everyone can find what is happening around a service.
  6. 6) Encryption - Ensuring encryption is used as part of API transport, and SSL is default for all services.

Using a single subdomain that can be used to access this resource.

Deployment

The deployment of any service, and defining how it can be delivered into a production environment.

  1. 1) Compute - The compute that is delivering each service.
  2. 2) Framework - Which programming framework is used to deploy each service.
  3. 3) Function - Which function is used to deploy each service.
  4. 4) Containers - Which container is used to deploy each service.
  5. 5) Gateway - Which gateway is used to deploy each service.
  6. 6) Pipeline - Information regarding the pipeline in use for service.
  7. 7) Region - Information regarding which region each service is deployed.

Providing a strong framework for consistently delivering services across an organization.

Orchestration

The orchestration of the build and deliver of a service, defining how the technical building blocks are orchestrated to deliver the resulting service.

  1. 1) Containers - The orchestration and management of containers, allowing APIs to be evolved, scaled, and managed in an automated way.
  2. 2) Build - Information regarding the build of each service and how it is deployed.
  3. 3) Hooks - Information regarding pre or post commit hooks for each service.
  4. 4) Jobs - Information regarding the jobs used as part of each service.
  5. 5) Events - Information about events used for managing jobs for each service.
  6. 6) Schedule - Information about the schedule sin use to manage each service.

Dependencies

Defining the dependencies associated with each service, helping quantifying what each service needs to do what it does successfully.

  1. 1) Service - Information about services each service depends on.
  2. 2) Software - Information about software each service depends on.
  3. 3) Data - Information about data each service depends on.
  4. 4) People - Information about people each service depends on.
  5. 5) Organization - Information about organization each service depends on.
  6. 6) Applications - Information about applications that depend on services.

Testing

The definitions for each service.

  1. 1) Scenarios - The scenarios for testing each service.
  2. 2) Saved Requests - Save specific requests from a service to be used in testing.
  3. 3) Results - The results of assertions tested against each service.

Performance

Establishing what the benchmarks are for a service, and what can be expected when it comes to performance.

  1. 1) Tests - The performance tests for each service.
  2. 2) Results - The results of performance test for each service.

Security

The definitions for each service.

  1. 1) Overview - The overview of security practices for each service.
  2. 2) Policies - The policies in use for each service.
  3. 3) Tests - The security tests in use for each service.
  4. 4) Results - The results of security for each service.

Here are the communication channels provided to you, helping you stay in tune with what is changing with each service, and the overall platform behind it.

  1. 1) Blog - The blog for each service.
  2. 2) Twitter - The Twitter account for each service.
  3. 3) Github - The Github account for each service.
  4. 4) Internal - Internal communication channel for each service.
  5. 5) External - External communication channels for each service.

These channels should provide you with a regular heartbeat of the changes being made, and what is being planned as part of this platform's operations.
